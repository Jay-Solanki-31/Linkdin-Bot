# Linkdin-Bot
# Architecture Version: v2 (BullMQ + Header-based URN Extraction)


This repository contains a LinkedIn content automation system (backend + frontend). The backend is an Express API that fetches external content, generates posts via an AI module, schedules posts into publish slots, and publishes to LinkedIn using BullMQ-based background workers. The frontend is a small Vite/React admin UI used for monitoring and triggering flows.

**Overview & Architecture**

- Backend: Node.js + Express API (entrypoint: `backend/src/server.js`) that exposes REST endpoints for fetchers, AI generation, slot allocation, publisher actions, LinkedIn OAuth, and a small dashboard for BullMQ.
- Background processing: BullMQ (Redis-backed) queues and workers located under `backend/src/queue` and `backend/src/queue/workers`. A Bull board dashboard is included in `backend/src/dashboard/bullboard.js`.
- AI module: located in `backend/src/modules/ai/ai.service.js` (referred to as the AI provider — Gemini/other provider integration lives in `backend/src/services/gemini.js`).
- Publisher: LinkedIn publishing logic under `backend/src/modules/publisher/linkedin.publisher.js` and `backend/src/services/linkedinToken.service.js` to manage access tokens.
- Database: MongoDB models live in `backend/src/models` and hold fetched content, generated posts and LinkedIn tokens.

End-to-end post lifecycle

The current implementation follows a scheduler‑driven, queue‑based workflow with explicit state recorded in MongoDB.

1. **Fetch** – a cron job (`modules/scheduler/fetchScheduler.js`) runs every Sunday at 11 AM IST (Asia/Kolkata) and/or can be invoked manually via `GET /api/:source`. It enqueues `FETCH_CONTENT` jobs on `fetcher-queue`, one per source. The `fetcher.worker.js` uses the adapters in `modules/fetchers` to pull articles and upserts them into the `FetchedContent` collection.

2. **Slot allocation & draft creation** – another Cron task (`modules/scheduler/slotAllocator.scheduler.js`) executes Mondays at 10 AM or when an operator POSTs to `/api/slot-allocator/run`. The `slotAllocator.worker.js` computes the next set of publish slots (hard‑coded in `SLOT_TIMES`), omits slots already taken by existing `GeneratedPost` documents, and for each free slot creates a new `GeneratedPost` draft with `status: "draft"` and a `publishAt` timestamp. Each draft immediately enqueues an AI job on `ai-processing-queue`.

3. **AI generation** – jobs on `ai-processing-queue` are processed by `ai.worker.js`. When a job runs it atomically marks the post `generating`, fetches the linked `FetchedContent`, and calls `modules/ai/ai.service.js` (Gemini prompt). On success the worker sets `text`, updates status to `queued`, and schedules a LinkedIn publish job with a delay computed as `publishAt - now`. Manual generation is available via `POST /ai/generate/:postId`.

4. **Publishing** – `linkedin.worker.js` pulls jobs from `linkedin-queue`; it ignores posts scheduled for the future or already `posted`. It marks the document `publishing`, calls `modules/publisher/linkedin.publisher.js` (which reads tokens from `linkedinToken.model.js`), and on success writes the returned URN and sets `status` to `posted`. Failures update the post to `failed` and increment `attempts`. Operators can manually queue a publish with `POST /api/publisher/ai-posts/:id/publish`.

5. **State & persistence** –
   - `FetchedContent` keeps raw items and uses a TTL index (`expiresAt`) so stale content is removed automatically.
   - `GeneratedPost` records lifecycle fields (`status`, `slot`, `publishAt`, `linkedinPostUrn`, `error`, `attempts`, etc.) and enforces a unique partial index on `linkedinPostUrn` to guard against duplicate publishes.

> All times are treated as UTC internally; API clients should use ISO‑8601 UTC timestamps (e.g. `2026-02-23T15:00:00Z`).

Notes:
- Internal state transitions live entirely in MongoDB models (`fetchedContent.model.js`, `generatedPost.model.js`).
- The dashboard (`backend/src/dashboard/bullboard.js`) exposes BullMQ job queues and statuses at `/admin/queues`.

Scheduling / `publishAt` behavior (UTC)

- All scheduling and `publishAt` timestamps are treated as UTC. When creating or updating `publishAt` values, provide UTC timestamps (ISO-8601). The scheduler and workers consider `publishAt` as absolute UTC time; the slot allocator also assigns UTC-based time slots.
- When passing `publishAt` via API, include timezone-neutral ISO strings, for example: `2026-02-23T15:00:00Z`.
- The `slotAllocator` may set a `publishAt` on `GeneratedPost`. The publish worker only attempts to publish when the job runs at or after the `publishAt` timestamp.

---

LinkedIn URN extraction from `x-restli-id` header

- Behavior: LinkedIn REST responses include an `x-restli-id` header that contains either an urn-like string or an ID. Typical values look like `urn:li:share:1234567890` or sometimes just `1234567890` depending on API and wrapper used.
- Implementation guidance (what the code does / should do):
  - Read the `x-restli-id` header from the LinkedIn publish response.
  - If the header value already starts with `urn:`, store it directly in the `linkedinPostUrn` field.
  - If the header is numeric or only an ID, convert it to an URN with the `urn:li:share:<id>` pattern before storing.

Example pseudocode:

```js
const header = response.headers['x-restli-id'];
const linkedinPostUrn = header && header.startsWith('urn:')
  ? header
  : header
    ? `urn:li:share:${header}`
    : null;
// Persist linkedinPostUrn to generatedPost
```

How the `linkedinPostUrn` unique partial index works

- Purpose: ensure that once a `GeneratedPost` has a persisted `linkedinPostUrn`, no other document can claim the same URN (prevents duplicate publishes) while allowing multiple drafts without a URN.
- Implementation pattern in MongoDB (example):

```js
db.generatedposts.createIndex(
  { linkedinPostUrn: 1 },
  { unique: true, partialFilterExpression: { linkedinPostUrn: { $exists: true } } }
);
```

- This is a partial unique index (or an equivalent sparse+unique alternative). It enforces uniqueness only for documents where `linkedinPostUrn` is present.

Retry and failure handling

- BullMQ job options: workers and queue producers set job options such as `attempts` and `backoff` policy to control automatic retries.
- On worker failure:
  - BullMQ moves a job into the failed state after retry exhaustion.
  - Failed job details are visible in the Bull dashboard (`backend/src/dashboard/bullboard.js`).
  - Application code should record per-post error details in `GeneratedPost` (e.g., `publishError`, `attempts`, `lastErrorAt`) so the UI and operators can inspect failures.
- Publish-specific handling:
  - If publish fails due to token expiration or transient LinkedIn errors, the job is retried according to job opts and token refresh is attempted via `linkedinToken.service.js`.
  - If the API returns an error that indicates a permanent failure (e.g., invalid content), workers should mark the `GeneratedPost` as `failed` and surface the error message.

How to run workers

Workers are individual Node scripts under `backend/src/queue/workers`. Each worker can be run directly (or managed by a process manager such as PM2, systemd, or Docker processes):

Example (run one worker):

```bash
cd backend
node src/queue/workers/fetcher.worker.js
node src/queue/workers/ai.worker.js
node src/queue/workers/linkedin.worker.js
node src/queue/workers/slotAllocator.worker.js
```

Recommendation for production:
- Use a process manager (PM2, Docker Compose, Kubernetes) to run one or more instances of each worker type. Keep `ai.worker` and `linkedin.worker` scaled separately depending on workload.
- Make sure your `REDIS_URL` and worker concurrency settings (in each worker file) are tuned for your environment.

Testing LinkedIn publishing via API

1. Confirm a `GeneratedPost` draft exists (via the frontend or by creating one with an AI generation endpoint). The generation endpoints live in `backend/src/routes/aiPosts.routes.js` or `ai.routes.js`.
2. Find the publish / publisher endpoint: `backend/src/routes/publisher.routes.js`.
3. Example `curl` to trigger a publish (adjust paths if your API is mounted under `/api`):

```bash
curl -X POST "http://localhost:3000/api/publisher/publish" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <YOUR_AUTH_TOKEN>" \
  -d '{ "generatedPostId": "<GENERATED_POST_ID>", "publishAt": "2026-02-23T15:00:00Z" }'
```

Notes:
- The exact request body and route params are defined in `backend/src/routes/publisher.routes.js` and handled by `publisher.controller.js`.
- Publishing will enqueue a job on the `linkedin.queue` which the `linkedin.worker.js` will pick up and attempt to post at/after `publishAt`.
